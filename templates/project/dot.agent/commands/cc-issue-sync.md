---
name: cc-issue-sync
description: Command - Sync GitHub Issues to a local TASKS.md file
---

## Prerequisites Check

1. Verify GitHub CLI is installed: `gh --version`
   - If not installed, inform user: `brew install gh` or see https://cli.github.com/
   - If missing, **stop** — do not attempt to fetch issues.

2. Verify authentication: `gh auth status`
   - If not authenticated, prompt user to run: `gh auth login`
   - If auth fails, **stop** — do not attempt to fetch issues.

3. Read `PROJECT.yaml` (if present in the project root):
   - Check `github_issues.enabled` — if the key exists and is `false`, warn the user:
     `github_issues.enabled is false in PROJECT.yaml — skipping issue sync.`
     Then stop unless the user passes `--force`.
   - Read `character` to determine grouping mode: `staged` or `toolbox` (default: `toolbox`).

## Fetch Issues

Run both fetches and capture results. If either `gh` call fails (network error, API error):
- Warn the user: `GitHub API unreachable — TASKS.md not updated. Existing file preserved as stale cache.`
- If TASKS.md already exists, add a warning comment at the top of the file:
  ```
  <!-- WARNING: sync failed YYYY-MM-DD HH:MM UTC — data may be stale -->
  ```
- Stop without writing further changes.

**Open issues:**
```bash
gh issue list --state open \
  --json number,title,labels,milestone,assignees,state,url \
  --limit 200
```

**Recently closed issues (last 30 days):**
```bash
gh issue list --state closed \
  --json number,title,labels,milestone,assignees,state,url,closedAt \
  --limit 50
```

Filter the closed list to issues closed within the last 30 days by checking the `closedAt` field.

## Determine Grouping Mode

Read `character` from `PROJECT.yaml`:

- **`staged`** — group by milestone (PoC / MVP / Production), then by status within each milestone.
- **`toolbox`** (default) — group by status label: `active`, `queued`, `idea`.

If `PROJECT.yaml` is absent, default to `toolbox` mode.

## Generate TASKS.md

Write the file at `TASKS.md` in the project root. Always overwrite the full file.

### Header (always present)

```markdown
<!-- auto-generated by cc-issue-sync — do not edit manually -->
# Tasks

> Last synced: YYYY-MM-DD HH:MM UTC
```

Use UTC time for the timestamp.

### No-issues case

If both open and recently-closed lists are empty, write:

```markdown
<!-- auto-generated by cc-issue-sync — do not edit manually -->
# Tasks

> Last synced: YYYY-MM-DD HH:MM UTC

No open issues.
```

Then stop.

### Issue entry format

Each issue is rendered as a list item:

```markdown
- [#42](https://github.com/owner/repo/issues/42) **Title of the issue** `label-a` `label-b` — @assignee
```

Rules:
- Issue number is a Markdown link using the `url` field.
- Title is bold.
- Each label name is wrapped in backticks as an inline code badge.
- Assignees: list GitHub logins prefixed with `@`, comma-separated. Omit if empty.
- Closed issues get a strikethrough title: `~~**Title**~~` and a `closed` badge appended.

### Staged project layout

Group open issues by milestone name. If no milestone is set, place in **Backlog**.
Within each milestone group, sub-group by status label:
- **In Progress**: issues with `status:active` label
- **Open**: all other open issues

Sort by issue number ascending within each sub-group.
Append a **Recently Closed** section at the end.

```markdown
## Milestone: PoC

### Open

- [#1](...) **Issue title** `label` — @assignee

### In Progress

- [#3](...) **Issue title** `in-progress` — @assignee

---

## Milestone: MVP

...

---

## Backlog

...

---

## Recently Closed

- [#7](...) ~~**Closed issue title**~~ `closed` — @assignee
```

Milestone ordering: sort milestones by their due date if available; otherwise alphabetically.
Place `Backlog` last, before `Recently Closed`.

### Toolbox project layout

Group open issues by status label. Recognised status labels (case-insensitive, strip `status:` prefix for grouping):
`status:active`, `status:queued`, `status:idea`.

Issues with no recognised status label go into **Uncategorised**.
Append a **Recently Closed** section at the end.

```markdown
## Active

- [#5](...) **Issue title** `active` `bug` — @assignee

---

## Queued

- [#9](...) **Issue title** `queued`

---

## Ideas

- [#12](...) **Issue title** `idea`

---

## Uncategorised

- [#2](...) **Issue title** `docs`

---

## Recently Closed

- [#4](...) ~~**Closed issue title**~~ `closed`
```

## Dry-Run Mode

If `--dry-run` is passed, generate the TASKS.md content and display it to the user
without writing the file. Report what would change compared to the existing TASKS.md
(if any). Useful for previewing sync results before overwriting.

## Usage Examples

```bash
# Sync issues using PROJECT.yaml settings
> /cc-issue-sync

# Force sync even if github_issues.enabled is false
> /cc-issue-sync --force

# Preview what would be written without modifying TASKS.md
> /cc-issue-sync --dry-run
```
