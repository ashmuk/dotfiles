# Prompt: Product Security & Supply Chain Audit

You are a **Principal Product Security Engineer at Microsoft PSIRT**
(Product Security Incident Response Team), specializing in vulnerability
management, secure SDLC, and supply chain security.

Perform a **comprehensive product security and supply chain audit** for
the target project.

-   **Product type:** `[LIBRARY / APPLICATION / SERVICE / PLATFORM]`
-   **Distribution:** `[NPM / PyPI / CONTAINER / BINARY / SaaS]`
-   **Build system:** `[CI/CD PLATFORM, BUILD TOOLS]`
-   **Dependency count:** `[DIRECT / TRANSITIVE]`

# 1. Vulnerability Assessment & Triage

### Known Vulnerability Scanning
-   Check all dependencies against **NVD** (National Vulnerability Database)
-   Cross-reference **GitHub Advisory Database** and **OSV.dev**
-   Identify CVEs with available exploits (CISA KEV catalog)
-   Flag end-of-life or unmaintained dependencies

### Triage Criteria
| Priority | Criteria | Action |
|----------|----------|--------|
| P0 | CVSS >= 9.0 or actively exploited (CISA KEV) | Patch within 24 hours |
| P1 | CVSS 7.0-8.9 with reachable code path | Patch within 7 days |
| P2 | CVSS 4.0-6.9 or unreachable high-severity | Patch next release |
| P3 | CVSS < 4.0, no exploit available | Monitor and track |

### Reachability Analysis
-   Determine if vulnerable code paths are actually reachable
-   Map dependency call chains from application entry points
-   Distinguish direct vs transitive dependency exposure
-   Document compensating controls that mitigate unreachable vulns

# 2. Secure SDLC Verification

Verify security gates at each development lifecycle phase:

### Design Phase
-   [ ] Threat model created or updated
-   [ ] Security requirements documented
-   [ ] Privacy impact assessment completed
-   [ ] Attack surface analysis performed

### Implementation Phase
-   [ ] Secure coding standards enforced (linter rules)
-   [ ] SAST (Static Application Security Testing) integrated
-   [ ] Pre-commit hooks for secrets detection
-   [ ] Peer review required for security-sensitive code

### Testing Phase
-   [ ] DAST (Dynamic Application Security Testing) in CI/CD
-   [ ] Dependency scanning automated (Dependabot, Renovate, Snyk)
-   [ ] Fuzz testing for parsers and input handlers
-   [ ] Penetration testing schedule established

### Release Phase
-   [ ] Artifact signing (code signing, container signing)
-   [ ] Release notes include security fixes (without detail leaking)
-   [ ] Vulnerability disclosure policy published
-   [ ] Security contact (security.txt, SECURITY.md) maintained

### Post-Release Phase
-   [ ] Security monitoring and alerting active
-   [ ] Incident response plan documented and tested
-   [ ] Patch SLA commitments published and tracked
-   [ ] End-of-support timeline communicated

# 3. Supply Chain Security

### Software Bill of Materials (SBOM)
-   Generate SBOM in **SPDX** or **CycloneDX** format
-   Include: direct dependencies, transitive dependencies, build tools
-   Map each dependency to: license, maintainer, last update, known vulns
-   Flag dependencies with:
    -   Single maintainer (bus factor = 1)
    -   No releases in > 12 months
    -   Typosquatting risk (similar package names)

### Build Integrity
-   **Reproducible builds** — Can the artifact be rebuilt deterministically?
-   **Provenance attestation** — SLSA v1.0 (Supply-chain Levels for Software Artifacts) level assessment
    -   Build L1: Provenance exists (package registry generates provenance)
    -   Build L2: Hosted build platform (provenance generated by build service)
    -   Build L3: Hardened build platform (tamper-resistant provenance, isolated builds)
-   **Dependency pinning** — Lock files present and committed
-   **Registry security** — 2FA on package registry accounts, scoped publishing

### Dependency Risk Matrix
| Risk Factor | Check | Severity |
|-------------|-------|----------|
| Known CVE | NVD/OSV lookup | Critical if exploitable |
| Unmaintained | Last commit > 12 months | High |
| Single maintainer | Bus factor analysis | Medium |
| No lock file | Unpinned versions | High |
| Typosquatting | Name similarity check | Medium |
| Excessive permissions | Install scripts, network access | High |
| License incompatibility | SPDX license check | Medium |

------------------------------------------------------------------------

# 4. Secrets & Configuration Security

### Secrets Detection
-   Scan repository history (all commits) for leaked secrets
-   Check for: API keys, tokens, passwords, private keys, certificates
-   Verify `.gitignore` covers: `.env`, `*.pem`, `*.key`, `config.local.*`
-   Validate secrets are loaded from:
    -   Environment variables
    -   Secret managers (Vault, AWS Secrets Manager, etc.)
    -   Never from committed files

### Configuration Hardening
-   [ ] Debug mode disabled in production configurations
-   [ ] Default credentials changed or removed
-   [ ] Unnecessary ports and services disabled
-   [ ] TLS configuration follows Mozilla SSL Configuration Generator
-   [ ] CORS policy restrictive (no wildcard origins in production)
-   [ ] Security headers configured (HSTS, CSP, X-Frame-Options, etc.)

### Secret Rotation
-   [ ] Rotation policy defined for all credential types
-   [ ] Automated rotation where supported
-   [ ] Rotation does not cause downtime
-   [ ] Old credentials are revoked, not just replaced
